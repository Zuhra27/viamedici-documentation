    VersionService, Document history

    General information

    Since the Data Store needs JSON data within the InMemory-Engine of EPIM, the following objects need to be versioned:

    • Hierarchies

    • Products

    • Variants

    • Images

    • Graphics

    • Documents

    • Audios

    • Videos

    • Texts

    • Editorial tables

    • Product tables

    To version these objects as simple and effective as possible, a service was implemented which only needs to be fed with item lists. Thus, versioning of objects can be easily added to existing and new PDK processes.

    Benefit:

    The versioning is always consistent with a central logic and central settings.

    Implementation

    The service consists of two job chains:

    • Feeder job

    • Processing job

    Feeder job

    The input file of the feeder job is an item list. The feeder determines the corresponding objects in EPIM. The following options are currently available:

    Type

    Description

    Hierarchy

    All child objects (groups, products and variants) of the hierarchy/subhierarchy are determined.

    Here you can differentiate between simple direct versioning and versioning with attributes and values within product relationships.

    Products/Variants

    All variants below a product are determined or, in the case of a variant, only the presence of the variant in EPIM is checked.

    As for the hierarchy, you can also differentiate between simple versioning and versioning with attributes and values in product relationships.

    Elements

    All elements are checked for presence in EPIM.

    VersionService, Feeder job


    The feeder always determines the internal EPIM Id for the objects and also adds the so-called epobType, which is necessary for creating a version.

    The final list of objects is transferred to the processor as an item list:

    Processing job

    Based on the input parameters, either a FULL or a DELTA is executed when processing the data. In the case of a DELTA, the timestamps in EPIM are compared:

    • Change data in EPIM > Creation date of current version

    The creation of the versions is done within this job using the web service daemon. The job chain and the corresponding processing job can be executed simultaneously:

    Feed structure

    The feeder can be called up with different "structures", depending on the version scenarios. The currently supported structures are described within the following subchapters.

    Note:

    To use the feeder, you need to create an item list XML file and copy it to the define VMHANDOVER directory. Please ensure that the file ending is .feed!

    HierarchyId

    Use the Id of a hierarchy to version the node itself as well as all of its child objects (groups, products and variants):

    [filename].feed
    <items processMode="delta" versionType="simple" versionNode="master" contentMode="viaPRODUCT" identifierType="hierarchyId" comment="my lovely content" languages="deu_DEU,fra_FRA">
    <item identifier="2" />
    <item identifier="53" />
    <item identifier="1065" />
    </items>

    Identifier

    Description

    processMode

    Use one of the following modes: full|delta.

    versionType

    simple: Regular versioning of the objects.

    withRelationData: Product relation attributes and values are included.

    withOrigin: Origin objects of references are included.

    versionNode

    Define the desired EPIM node on which the versioning should be processed (see chapter multi-node configuration).

    If empty, the master node will be automatically used.

    contentMode

    You have to use viaPRODUCT.

    identifierType

    You have to use hierarchyId.

    identifier

    Create an item with it for each hierarchy node, use the Id as the identifier.

    comment

    Define specific content (this will overwrite the comment loaded from the process-config.xml).

    languages

    Languages in which the versions are to be created. If nothing is defined, versioning will be done in all versioning languages configured in EPIM. Can be used only in conjunction with the use of the WebService (queueType="webService").

    VersionService, Configuration

    Versioning of hierarchy and product elements with the consideration of the workflow status

    In the past, when versioning products, the entry for versioning the product was written into the table "db_version_river" and the trigger (force_elements) was set, so that EPIM would version the corresponding elements continuously. Unfortunately, EPIM does not take the workflow status when versioning the elements.

    Therefore, the versioning service was changed in such a way that it does not set the trigger (force_elements) anymore (setting in the Config), but searches out the associated elements, checks the workflow status and makes an own entry for the elements into the table "db_version_river".

    Multi-node configuration

    Follow these steps to configure a feed and the processing for an additional EPIM/PDK node:

    1. Add the configuration of the new node to the ${customer}-config-${location}.xml.

    2. Enable the masterSlaveInUse option in process-config.xml.

    3. Define a code for the master node in process-config.xml.

    4. Add the versionNode information into the .feed files.

    5. Configure an additional agent on the PDK (JobScheduler).

    6. Adapt the process "260_NA_versioning_service" and deploy the adaption on the slave node.

    7. Create a new node-specific lock for the job chain.

    8. Extend the PDK process "210_versioning_service_FEEDER" and add slave specific jobs to the chain.

    9. Test your configuration with a .feed file.

    PDK agent

    Create a new PDK agent with an XML like this within the folder of the job chain:

    my_name.process_class.xml
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <process_class max_processes="10" remote_scheduler="http://40.117.90.118:4445"/>

    This new PDK agent will later be used in the node-specific job chains and jobs.

    Here the PDK agent needs to be added to the job element, by doing this the master scheduler will run the job on the defined agent:

    Add agent to a job chain
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <job_chain orders_recoverable="yes" visible="yes" title="version service" max_orders="10" process_class="01_ProcessClass_Agent_2">
    Add agent to a job
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <job title="Feeding logik" order="yes" stop_on_error="no" tasks="10" java_options = "-Xmx128m" process_class="01_ProcessClass_Agent_2">

    Adaption of the process 260_NA_versioning_service

    Use the file "260_NA_processing_versions.actionlist.xml" as a base. In the first lines of the actionList the local, node-specific properties are defined. Use the already configured node-specific properties of the customer config file to initialize the new local properties:

    260_NA_processing_versions.actionlist.xml
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
    <actionList xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" logLifetime="2" logLevel="-9" title="processing versions" xsi:noNamespaceSchemaLocation="./org.vm.actionlist.ActionList.xsd">
    <actions>
    <property name="naProcessBackupDir" value="${NA_VMBACKUP}/${processName}" handout="true" overwrite="true" title = "set processBackupDir"/>
    <property name="naSessionBackupDir" value="${NA_VMBACKUP}/${processName}/${sessionId}" handout="true" overwrite="true" title = "set sessionBackupDir"/>
    <property name="naSessionWorkDir" value="${NA_VMWORK}/${processName}/${sessionId}" handout="true" overwrite="true" title = "set sessionWorkDir"/>
    <property name="naSessionLogfile" value="${naSessionWorkDir}/logfile.txt" handout="true" overwrite="true" title = "set sessionLogfile"/>
    <mkDir dir="${naProcessBackupDir}" title="create sessionWorkDir"/>
    <mkDir dir="${naSessionBackupDir}" title="create sessionWorkDir"/>
    <mkDir dir="${naSessionWorkDir}" title="create sessionWorkDir"/>

    Tip:

    Rename the properties and a prefix, like the prefix in the customer config.

    Search and replace the names in the actionList.

    This prevents overwriting in multiple processing sessions.

    Finally adjust the job chain:

    • Name of job chain.

    • Jobs of job chain.

    • Name of actionLists.

    • PDK agent.

    • etc.

    Then deploy it on the master PDK as a new Job Chain.

    New node-specific lock

    To prevent overwriting of files during the processing of the .feed files in an additional node, a lock must be added to the process/processing jobs.

    These locks must be node-specific, so that the nodes are working independently and do not have to wait for another session on a different node to be finished first.

    Therefore, create a XML like this within the folder of the job chain:

    my-name.lock.xml
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <lock />

    This new lock will later be used in the node-specific jobs. Here the lock needs to be added as a new element within the job element:

    Lock in a job
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <job title="Feeding logik" order="yes" stop_on_error="no" tasks="10" java_options = "-Xmx128m" process_class="01_ProcessClass_Agent_2">
    <lock.use lock="my-name" exclusive="yes"/>

    Extend the process 210_versioning_service_FEEDER

    To enable the new node in the feeder, you need to add new jobs to the job chain.

    MUST:

    • The state of the first new job in the job chain needs to have state: 220_[code of node].

    • In this example NA is used as code for the new node.

    • This code must be used in the .feed file as versionNode, otherwise the .feed file will not be processed.

    • Info: The regular job "210_feeder" will extract the value of the versionNode, if the value is not equal to the configured code of the master, then the job will call the job with the state "220_${identifyNode}"; identifyNode is initialized with this XPath: /items/@versionNode.

    Node specific job in the job chain
    <?xml version="1.0" encoding="ISO-8859-1"?>
    <job_chain orders_recoverable="yes" visible="yes" title="version service FEEDER" max_orders="10">
    <file_order_source directory="/opt/viamedici/VMHANDOVER/datastore-v1.0/versionService" regex=".*\.feed$" max="10" />
    <job_chain_node state="110" job="110_initialize" next_state="210" error_state="940"/>
    <job_chain_node state="210" job="210_feeder" next_state="220_NA" error_state="940"/>
    <!-- Feed in EPIM NA -->
    <job_chain_node state="220_NA" job="220_NA_transfer_inbox" next_state="230_NA" error_state="940"/>
    <job_chain_node state="230_NA" job="230_NA_feeder" next_state="900" error_state="940"/>
    <job_chain_node state="900" job="900_finalize-success" next_state="980" error_state="940"/>
    <job_chain_node state="940" job="940_finalize-error" next_state="980" error_state="980"/>

    Finally adjust the new jobs:

    • State and name of job chain.

    • Name of actionLists.

    • PDK agent.

    • Lock.

    • etc.

    Then deploy it on the master PDK.
